- P0：安全与数据保护（规则7行分隔符、规则11删除保护）
- P1：工具选择与执行路径（规则1文件操作、规则2命令行、规则3 Python、规则17路径解析、规则18替换前读取）
- P2：操作规范与流程（规则6邮件通知、规则9 git规范、规则10禁止自动提交、规则12 git push、规则14脚本与文档语言、规则16大文件分批操作、规则19 Markdown标题序号）
- P3：输出与展示（规则4输出优化、规则5执行确认、规则8输出格式、规则13乱码处理、规则15浏览器）

**详细规则：**

- 1.文件操作优先级（P1）：优先内置文件工具读写；失败重试最多10次；仍失败才降级Python代码操作
- 2.命令行执行（P1）：
  - 强制使用MCP `start_process`/`interact_with_process`，严禁内置`executeBash`，覆盖所有命令行场景
  - 乱码预防：优先用Python `subprocess`（`encoding='utf-8'`）执行，禁止PowerShell/CMD内联处理需解析的输出
  - Windows带参数命令（含空格/引号/特殊字符，如git commit -m）：必须直接用Python `subprocess`列表传参（如`['git','commit','-m','msg']`），禁止先尝试CMD/PowerShell再降级
  - 重试上限：同一命令失败或乱码，重试不超过2次，超过则停止告知用户
- 3.Python操作（P1）：优先使用MCP工具执行Python代码
- 4.MCP输出优化（P3）：解析原始输出→提取关键信息→用简洁中文描述（成功/失败、操作内容、重要数据/错误）；严禁显示原始JSON/转义字符串；有错误须说明原因和建议
- 5.执行确认（P3）：确认MCP工具执行真正成功；异常则输出失败结果并尝试其他方式
- 6.邮件通知（P2）：
  - 触发：文件修改、代码执行、配置变更、文档生成、git写操作（commit/merge/rebase/cherry-pick等）
  - 不触发：只读操作（git status/log/diff等）
  - 顺序：完成操作→【强制检查点】自问"是否已发送邮件？"→发送邮件→输出结果
  - 失败重试1次，仍失败记录错误并继续
- 7.行分隔符（P0）：写入前必须将\r\n替换为\n；写入后检查并转换（`content.replace(b'\r\n', b'\n')`）；零容忍CRLF
- 8.输出格式（P3）：每次MCP工具执行后必须按以下多行格式输出，每项独占一行：
  ```
  [状态] √ 执行成功 / × 执行失败
  [操作] [简洁描述主要操作]
  [结果] [提取的重要信息]
  [警告] [如有错误或警告]
  ```
- 9.git提交规范（P2）：格式`<type>: <description>`，50字符内，祈使语气，首字母小写，无句号，强制英文；类型：feat/fix/docs/refactor/perf/test/chore/style/ci/revert
- 10.禁止自动提交推送（P2）：执行完成后禁止自动git commit，禁止自动git push
- 11.删除保护（P0）：untracked文件/目录必须先git commit再删除，禁止直接删除
- 12.git push（P2）：优先使用MCP工具（DesktopCommanderMCP）
- 13.乱码处理（P3）：命令行输出乱码→立即切换Python `subprocess`重试（见规则2）；禁止在CMD/PowerShell上反复重试；上限2次，超过则停止报告失败
- 14.脚本与文档语言（P2）：脚本文件(.py/.sh/.bat/.cmd/.ps1)严禁中文内容，Python需UTF-8编码声明；md文档在用户无特殊语言要求时默认使用中文；其他文件类型不受限
- 15.浏览器自动化（P3）：必须通过MCP工具
- 16.大文件分批操作（P2）：
  - 原则：分步修改，每步只改一个区域并验证；全部完成后检查结构完整性、逻辑连贯性、格式统一性
  - 触发：预估文件超200行，或单次替换超50行，或多区域修改
  - 新文件：声明分步计划→创建文件写入第1模块→追加后续模块（每步≤100行）→每步读取验证→完成后语法检查
  - 改文件：声明分步计划→每步只替换一个区域（每步≤50行）→每步读取验证→批量替换先声明所有替换点再逐一执行→完成后检查一致性
  - 降级：中文匹配失败超2次→立即降级为文件追加重写或Python脚本替换，禁止同一匹配重试超2次
  - 禁止：单次创建超200行；因文件大而放弃/截断；批量替换后不验证
  - 模块划分：按功能边界拆分，每步有明确模块名称
  - 失败恢复：重试当前步，不影响已完成步骤
- 17.文件路径解析（P1）：
  - 问题：IDE的relativeFilePath可能含工作区文件夹名（如根为`d:\docs\tech`，相对路径为`tech\sap\file.md`），盲拼导致重复
  - 强制流程：取工作区根→取相对路径→检查相对路径首段是否与根目录末段相同→相同则去掉首段再拼接，不同则直接拼接
  - 禁止不经检查直接拼接；禁止路径错误后才用fileSearch补救（应一次成功）
  - 兜底：解析后仍不存在→fileSearch搜索文件名，限1次
- 18.内容替换前置读取（P1）：
  - 问题：对话上下文显示的内容可能因编码/渲染/截断与实际不一致，凭记忆构造oldStr会匹配失败
  - 强制流程：替换前先fsRead目标文件（刚读取过且未修改可复用）→从读取结果精确复制原文作为oldStr→禁止从记忆/推测/缓存构造
  - 特别注意：中文相似字（则/刹、已/己）、全角/半角数字（１８ vs 18）、全角/半角标点极易混淆，以读取结果为准
  - 禁止不读取就替换；禁止失败后才读取确认（应一次成功）
- 19.Markdown标题序号风格（P2）：
  - 强制：所有生成的Markdown文档标题序号必须使用纯数字风格，如`1.`、`1.1`、`1.1.1`
  - 禁止：中文数字序号（一、二、三）、混合风格（如一级用中文「二、」二级用数字「2.1」）
  - 示例：`# 2. 具体问题详述` → ✓；`# 二、具体问题详述` → ✗
