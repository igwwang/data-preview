# 需求分析模板使用指南

> 本文档是 requirements-analysis-template-chinese.md（需求分析模板）的配套指南，说明模板的设计背景、使用流程和推荐提示词。

## 1. 为什么需要需求分析模板

### 1.1 问题

AI 编程的核心链路是：**需求分析 → 架构设计 → 代码实现 → 测试**。

已有架构设计模板（architecture-design-template-chinese.md）规范了架构设计的输出，但如果上游的需求分析是非结构化的、模糊的，AI 在生成架构设计时就会"脑补"，质量不可控。

### 1.2 模板的本质

**约束 AI 的输出边界，减少幻觉。** 给 AI 一个提问清单和质量标准，确保收集到的信息足够完整，下游不用猜。

### 1.3 "需求多种多样，难以通用"的顾虑

这个顾虑合理，但互联网大厂的实践证明可以解决。关键思路是**分层模板，而非单一模板**：

| 层级 | 内容 | 稳定性 |
|------|------|--------|
| 通用骨架层 | 目标、用户、场景、约束、验收标准 | 稳定，所有项目通用 |
| 领域扩展层 | 按项目类型选填（ToB/ToC、新建/重构、移动/数据/AI等） | 可变，按需裁剪 |

### 1.4 行业最佳实践

| 公司 | 方法 | 核心思路 |
|------|------|----------|
| Google | PRD + Design Doc 两阶段 | PRD 聚焦 What/Why，Design Doc 聚焦 How，PRD 有标准模板但允许裁剪 |
| Amazon | Working Backwards（逆向工作法） | 从 Press Release + FAQ 开始，强制先想清楚用户价值，再展开细节 |
| 字节/阿里 | 通用 PRD + 领域补充模块 | 如电商有交易流程模块，社交有内容安全模块 |

共同点：**核心骨架固定，细节按需扩展**。

## 2. 模板设计思路

### 2.1 结构

采用"通用骨架 + 可选模块"：

- **必填骨架（第1-7章）**：项目概述、目标用户、核心场景、功能需求、非功能需求、约束条件、验收标准
- **选填扩展（第8章）**：移动应用 / 数据AI / 系统重构 / ToB企业级 / 前端交互

### 2.2 与架构设计模板的衔接

需求分析文档的输出直接映射到架构设计模板的输入：

| 需求分析模板章节 | → | 架构设计模板对应章节 |
|-----------------|---|---------------------|
| 1. 项目概述 | → | 1.1 业务背景 |
| 1.5 技术可行性验证 | → | 2.1 设计原则和技术选型（POC结果作为选型依据） |
| 2. 目标用户 | → | 1.2 使用场景 |
| 3. 核心场景 | → | 3. 核心业务流程 |
| 4. 功能需求 | → | 1.3 核心需求 + 2.4 服务拆分 |
| 4.3 数据需求 | → | 2.5 数据架构设计 |
| 4.4 接口需求 | → | 2.4 服务拆分（接口设计）|
| 5. 非功能需求 | → | 1.3 质量属性 + 5. 性能与容量规划 |
| 6. 约束条件 | → | 1.4 约束条件 + 2.1 设计原则和技术选型 |
| 7. 验收标准 | → | 11. 测试策略 |

## 3. 使用流程

### 3.1 完整工作流

```
用户粗略想法
  → 立项/预研确认（方向判断、技术选型初筛、资源可行性）
  → AI 按需求模板引导提问、补充细节
  → 生成完整需求分析文档（含1.5 POC验证）
  → 【POC验证通过】
  → AI 读取需求文档 + 架构模板
  → 生成架构设计文档
  → 代码实现
  → 测试
```

> **关键关卡**: 立项/预研确认未通过不开始需求分析；POC验证未通过不进入架构设计。

### 3.2 两种典型用法

**用法1：AI 引导式（推荐）**

适用于只有粗略想法、写不出完整需求文档的场景（最常见）。

采用**"1轮对话 + 问卷文档"两步走**模式：

**Step 1: 方向性对话（1轮）**
1. 用户给出粗略需求描述（可能只有几句话）
2. AI 读取模板，提出少量关键的方向性问题（如：做什么、给谁用、技术偏好、核心约束）
3. 目的是让 AI 获得足够上下文，以便生成高质量的问卷

**Step 2: 问卷文档采集（异步）**
1. AI 基于 Step 1 的理解，生成覆盖所有章节的完整问卷文档
2. 已知信息预填充，未知信息列为问题并提供选项参考（a/b/c 降低填写成本）
3. 用户在文档中按自己节奏填写（可分批）
4. AI 读取问卷，澄清矛盾和风险点（1-2轮对话）
5. AI 生成完整需求分析文档

**为什么不逐章提问？**

逐章对话模式（每次问一个章节，等回答后继续下一个）存在明显效率问题：
- 对话轮次多（通常需要7-8轮），用户等待时间长
- 用户无法预览全部问题，难以统筹回答
- 前面章节的回答可能影响后面的问题，但逐章模式下无法提前感知

问卷文档模式的优势：
- 用户可以一次性看到所有问题，按自己节奏填写
- 已确认内容汇总在文档顶部，方便回顾和校对
- 对话轮次从7-8轮缩减到3-4轮
- 问卷本身也是过程文档，便于团队协作和追溯

**问卷文档设计要点**（详见 requirements-questionnaire-template.md）:
- 每个问题提供 2-4 个选项参考（a/b/c/d），降低用户填写成本
- 已知信息预填充并标记 ✅，待回答问题标记 ❓
- 按模板章节组织，方便用户定位
- 每个问题下方独立一行 `回答：` 后留空，统一回答区域格式

AI 的价值在于：
- 知道**该问什么**（模板定义了结构）
- 知道**问到什么程度够了**（每个字段有明确标准）
- 知道**缺什么会导致架构设计脑补**（映射关系表明确了依赖）
- 能在生成文档前**主动发现矛盾和风险点**并澄清

**用法2：用户主导式**

适用于已有较完整资料（PRD/会议纪要/竞品分析等）的场景。

1. 用户按模板结构准备好需求文档，或提供已有资料
2. AI 读取后提取信息填充模板，做完整性检查，指出缺失项
3. 用户补充后，AI 确认可以进入架构设计阶段

## 4. 推荐提示词

### 4.1 从零开始（只有粗略想法）— 推荐

```
我想做[一句话描述需求]。

请参考 @requirements-analysis-template-chinese.md 模板，帮我完善需求。
规则：
1. 先问我几个关键的方向性问题（不超过5个），了解项目大方向
2. 基于我的回答，生成一个完整的问卷文档（覆盖模板所有必填章节），已知信息预填充，未知信息列为问题并提供选项
3. 我在文档中填写后，你读取并澄清矛盾点
4. 最终输出完整的需求分析文档
```

### 4.1b 从零开始（备选：逐章提问模式）

适用于问题较少（预估<10个）或用户偏好对话式交互的场景。

```
我想做[一句话描述需求]。

请参考 @requirements-analysis-template-chinese.md 模板，通过逐步提问帮我完善需求。
规则：
1. 每次只问一个章节的问题，等我回答后再继续下一个
2. 我回答后，直接填充到模板对应章节
3. 所有必填章节（第1-7章）完成后，检查完整性并指出缺失项
4. 最终输出完整的需求分析文档
```

### 4.2 已有部分资料

```
我想做[一句话描述需求]，已有相关资料如下：
@已有文档1.md
@已有文档2.md

请参考 @requirements-analysis-template-chinese.md 模板：
1. 先从已有资料中提取信息，填充模板能覆盖的部分
2. 列出还缺少哪些关键信息，按优先级向我提问
3. 我补充后继续填充，直到必填章节完成
4. 最终输出完整的需求分析文档
```

### 4.3 提示词设计要点

| 要点 | 说明 |
|------|------|
| 指定模板 | 让 AI 知道输出结构，避免自由发挥 |
| 明确交互方式 | 推荐问卷文档模式；逐章提问仅作为备选 |
| 明确完成标准 | 必填章节全部完成 + 完整性检查通过 |
| 方向性对话先行 | 问卷模式下，先用1轮对话建立上下文，再生成问卷 |

## 5. 实践经验

> 以下经验来自 Tool Carousel 项目的需求分析实践，详见 `docs/requirements/process-summary.md`。

### 5.1 模式选择决策树

```
用户给出初步需求
  ↓
AI 评估：预估剩余问题数量
  ↓
≤ 10个问题 → 逐章提问模式（4.1b）
> 10个问题 → 问卷文档模式（4.1，推荐）
```

### 5.2 问卷文档模式的关键节点

| 节点 | 要点 |
|------|------|
| 方向性对话 | 控制在1轮，只问大方向（做什么、给谁、技术偏好），不深入细节 |
| 问卷生成 | 一次性覆盖所有章节；已知信息预填充；提供选项降低填写成本 |
| 用户填写 | 允许分批填写；AI每次读取后确认新增内容 |
| 矛盾澄清 | 生成文档前必须完成；重点关注：逻辑矛盾、潜在风险、模糊表述 |
| 文档生成 | 大文档分步写入，每步验证 |

### 5.3 常见陷阱

| 陷阱 | 说明 | 规避方式 |
|------|------|----------|
| 逐章提问陷入长对话 | 问题多时对话轮次膨胀，用户疲劳 | 预估问题>10个时切换问卷模式 |
| 问卷问题质量低 | 没有足够上下文就生成问卷，导致问题不相关 | 先用1轮对话建立上下文 |
| 跳过矛盾澄清 | 直接生成文档，事后发现矛盾需返工 | 生成前强制检查一致性 |
| 已确认信息丢失 | 多轮对话后早期信息被遗忘 | 问卷顶部汇总已确认内容 |
| 缺少理解确认 | AI误解需求后直接生成大量输出，全部返工 | 生成文档前先输出3句话理解摘要，等用户确认 |
| 跳过POC验证 | 在错误的技术方向上完成全流程后才发现不可行 | 第1章完成后强制执行1.5节POC验证 |
