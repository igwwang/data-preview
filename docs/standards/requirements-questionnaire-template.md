# 需求问卷调查模板

> **定位**: 配合 requirements-analysis-template-chinese.md 使用，规范 AI 生成问卷的结构和风格。
>
> **使用场景**: AI 引导式需求分析（用法1）中，Step 2 生成问卷文档时，按此模板输出。

---

## 1. 模板结构

```markdown
# [项目名称] - 需求分析问卷

> **填写说明**:
> - ✅ = 已确认，无需填写
> - ❓ = 待回答，请在 **回答：** 后面填写选项字母或自由描述
> - 填完后告诉我，我会读取并澄清矛盾点

---

## 已确认信息汇总

| 项目 | 确认内容 |
| ---- | -------- |
| ...  | ...      |

---

## 立项/预研确认

> 对应模板“立项/预研确认”章节。已在方向性对话中确认的项目标记 ✅，未确认的列为问题。

---

## 第N章 [章节名称]

### N.x [小节名称]

✅ **[标签]**: [已确认的内容]

❓ **Q[编号]**: [问题描述]
- a) [选项A]
- b) [选项B]
- c) [选项C]

回答：

---

> 填完后告诉我，我会读取并进行矛盾澄清，最终输出完整的需求分析文档。
```

---

## 2. 格式规则

### 2.1 文件命名

```
docs/requirements/requirements-questionnaire.md
```

### 2.2 标记符号

| 符号 | 含义 | 使用场景 |
| ---- | ---- | -------- |
| ✅   | 已确认 | 从用户输入或对话中已明确的信息 |
| ❓   | 待回答 | 需要用户补充的问题 |

### 2.3 问题编号

- 全局连续编号：Q1, Q2, Q3...
- 不按章节重置编号
- 便于用户快速定位和引用

### 2.4 回答区域

- 每个问题下方独立一行 `回答：` 后留空
- 不使用内联占位符（如 `【回答】`、`___`）
- 不在选项中嵌入填写位置

```markdown
<!-- ✅ 正确 -->
❓ **Q1**: 问题描述？
- a) 选项A
- b) 选项B

回答：

<!-- ❌ 错误 -->
❓ **Q1**: 问题描述？
- a) 选项A
- b) 选项B
- c) 其他：`【回答】`
```

### 2.5 选项设计

- 每个问题提供 2-4 个选项，降低填写成本
- 选项用小写字母 a/b/c/d 编号
- 选项内容简洁，括号补充说明
- 不设"其他"选项，用户可在 `回答：` 处自由描述

```markdown
<!-- ✅ 正确 -->
❓ **Q6**: 默认轮播间隔时间？
- a) 30 秒
- b) 60 秒
- c) 120 秒

回答：

<!-- ❌ 错误：选项过多、描述冗长 -->
❓ **Q6**: 您期望的默认轮播间隔时间是多少秒？请根据实际业务场景选择：
- a) 15 秒（适合快速浏览）
- b) 30 秒（适合一般监控）
- c) 45 秒（适合详细查看）
- d) 60 秒（适合大屏展示）
- e) 90 秒
- f) 120 秒
- g) 其他：请填写___
```

### 2.6 已确认信息

- 用 ✅ 加粗标签标记
- 标签简短：现状、核心目标、预填充 等
- 表格形式汇总多条已确认信息

```markdown
✅ **现状**: 团队需要在大屏上展示多个监控 Dashboard。

✅ **预填充**:

| 维度 | 描述     |
| ---- | -------- |
| ...  | ...      |
```

## 3. 章节对应关系

问卷章节必须与 requirements-analysis-template-chinese.md 的必填章节一一对应：

| 问卷章节 | 对应模板章节 | 内容要点 |
| -------- | ------------ | -------- |
| 已确认信息汇总 | （无对应） | 方向性对话中已明确的所有信息 |
| 立项/预研确认 | 立项/预研确认 | 方向判断、技术选型初筛、备选方案排除、资源可行性、干系人共识 |
| 执行摘要 | 执行摘要 | 目标用户+核心问题+解决方案+预期收益，100字内 |
| 第1章 项目概述 | 1. 项目概述 | 背景、目标、范围、成功标准、POC |
| 第2章 目标用户 | 2. 目标用户分析 | 用户画像、用户旅程 |
| 第3章 核心场景 | 3. 核心场景与用户故事 | 场景列表、P0场景详述 |
| 第4章 功能需求 | 4. 功能需求 | 功能清单、数据需求、接口需求 |
| 第5章 非功能需求 | 5. 非功能需求 | 性能、稳定性、安全、兼容性 |
| 第6章 约束条件 | 6. 约束条件 | 技术约束、业务约束、风险 |
| 第7章 验收标准 | 7. 验收标准与交付定义 | 交付物、验收检查清单 |
| 第8章 领域扩展 | 8. 领域扩展模块 | 按项目类型选用 |

## 4. 内容生成规则

### 4.1 已确认信息汇总

- 将方向性对话中所有已明确的信息汇总到文档顶部
- 用表格形式，左列为项目维度，右列为确认内容
- 便于用户快速回顾和校对

### 4.2 问题生成原则

| 原则 | 说明 |
| ---- | ---- |
| 不重复已确认信息 | 已在对话中明确的内容标记 ✅，不再提问 |
| 聚焦决策点 | 只问影响架构设计和实现的关键问题 |
| 提供选项 | 每个问题给出常见选项，降低用户思考成本 |
| 按章节组织 | 问题归入对应的模板章节，便于定位 |
| 控制总量 | 问题总数建议 10-20 个，超过 20 个考虑分批 |

### 4.3 章节处理策略

| 信息状态 | 处理方式 |
| -------- | -------- |
| 已完全确认 | 整节标记 ✅，展示预填充内容 |
| 部分确认 | ✅ 标记已知部分，❓ 列出待确认问题 |
| 完全未知 | ❓ 列出该章节的关键问题 |
| 不适用 | 省略该章节，或标注"不适用" |

---

## 5. 质量检查清单

生成问卷后，AI 自检：

- [ ] 已确认信息汇总完整，无遗漏
- [ ] 问题编号全局连续（Q1, Q2, Q3...）
- [ ] 每个问题都有 `回答：` 留空行
- [ ] 选项数量 2-4 个，无"其他"选项
- [ ] 章节覆盖模板所有必填章节（立项/预研确认 + 第1-7章）
- [ ] 不适用的选填章节已省略或标注
- [ ] 问题不重复已确认信息
- [ ] 文档末尾有填写完成后的引导语
